# CogOps Vault Contract

The vault is the durable knowledge store for the CogOps epistemic engine. This document defines what each directory contains, does NOT contain, the belief artifact schema, and the status lifecycle.

## Directory Structure

```
vault/
  beliefs/        Durable system understanding
  verification/   Challenges to understanding
  actions/        Task outputs and reports
  evolution/      Skill specs, trials, promotions
    skills/       Individual skill packages
    registry.md   Master skill registry
  media/          Shared render assets only
```

## What Each Directory Contains

### beliefs/
**Contains:** Markdown files with YAML frontmatter describing machine-inferred understanding of source modules, architecture patterns, canonical flows, and system contracts.

**Does NOT contain:** Raw source code, test outputs, action results, verification reports, or evolution artifacts.

**Naming convention:** `{entity}_{claim_id_prefix}.md` for module beliefs; `arch_{pattern}_{unique_id}.md` for architecture beliefs; `cogops_{topic}_{claim_id_prefix}.md` for system-level documentation.

### verification/
**Contains:** Markdown files that challenge beliefs. Schema compliance checks, coverage gap analyses, staleness reports, graph connectivity audits, and epistemic layer audits.

**Does NOT contain:** Beliefs themselves, action outputs, or evolution artifacts.

### actions/
**Contains:** Task outputs produced by the Action layer. PR briefs, context packs, code generation results, reports, dashboards.

**Does NOT contain:** Beliefs, verification results, or evolution artifacts.

### evolution/
**Contains:** Skill specifications, trial results, promotion records, and the skill registry.

**Does NOT contain:** Beliefs, verification reports, or action outputs.

#### evolution/skills/{skill-id}/
Each skill package contains:
- `SKILL.md` -- Skill specification (intent, preconditions, steps, postconditions)
- `tool.py` -- Executable tool implementation
- `metrics.json` -- Performance metrics (success rate, latency, cost)
- `tests/` -- Test cases for the skill

### media/
**Contains:** Shared render assets (diagrams, images) referenced by beliefs or actions.

**Does NOT contain:** Source code, data files, or any non-render assets.

## Belief Artifact Schema

Every belief file MUST have this YAML frontmatter:

```yaml
---
claim_id: <unique identifier>
entity: <module or concept name>
status: <observed | inferred | verified | stale | hypothesis>
confidence: <0.0 to 1.0>
sources:
  - <file:line references to source code>
last_checked: <ISO 8601 timestamp>
derived_from:
  - <claim_ids or tool names that produced this belief>
epistemic_layer: <truth | belief | verification | action | evolution>
---
```

Optional fields for boundary files:
```yaml
boundary_note: "<resolution rule explaining the boundary>"
```

## Status Lifecycle

```
observed --> inferred --> verified --> stale --> hypothesis
    |                        |           |
    +--------<---------------+           |
    |        (re-verified)               |
    +--------<---------------------------+
             (re-observed)
```

1. **observed** -- Raw data captured from source code by Truth layer tools (SAST, depgraph, skeleton). No interpretation applied.

2. **inferred** -- Machine-compiled understanding. The [[belief_compiler_18a33f4c]] has processed Truth artifacts and generated a structured claim. This is the default status for all auto-generated beliefs.

3. **verified** -- A verification artifact in `vault/verification/` has challenged this belief and confirmed it. The [[verification_engine_18a33f4c]] promotes inferred beliefs to verified.

4. **stale** -- The source code has changed since `last_checked`, or the staleness threshold has elapsed. Stale beliefs trigger Flow 2 (Verify Before Answer) in the router.

5. **hypothesis** -- A speculative belief generated by the Evolution layer. Not yet confirmed by Truth or Verification. Used for experimental skill synthesis.

## Epistemic Layer Tags

Every belief file carries an `epistemic_layer` tag identifying which layer of the CogOps topology it belongs to:

- **truth** -- Reads reality (AST, deps, watchers, traces)
- **belief** -- Writes machine understanding
- **verification** -- Challenges understanding
- **action** -- Uses understanding
- **evolution** -- Improves machinery

Files straddling two layers use compound tags (e.g., `truth/belief`) and include a `boundary_note` explaining the resolution rule.
